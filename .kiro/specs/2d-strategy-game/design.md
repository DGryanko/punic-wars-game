# Дизайн документ: "Пунічні війни: Табори"

## Огляд

"Пунічні війни: Табори" - це 2D міні-RTS гра, розроблена на рушії Raylib, що фокусується на управлінні військовими таборами двох історичних фракцій: Риму та Карфагену. Гра поєднує класичні елементи RTS з унікальними механіками моралі та ресурсів, характерними для кожної цивілізації.

### Ключові особливості
- Дві асиметричні фракції з унікальними особливостями
- Система моралі, що впливає на боєздатність залежно від ресурсів
- Розміщення табору гравцем через систему "Обозу"
- Економіка на основі рабів та робітників
- Компактний розмір гри (~50-100 МБ) завдяки Raylib

## Архітектура

### Загальна структура
Гра побудована на модульній архітектурі з чіткими розділеннями відповідальності:

```
main.cpp (головний цикл гри)
├── building.h (система будівель)
├── unit.h (юніти та їх поведінка)
├── slave.h (раби та збір ресурсів)
├── morale.h (система моралі фракцій)
├── resources.h (управління ресурсами)
├── combat.h (бойова система)
└── ai.h (штучний інтелект)
```

### Raylib інтеграція
- **Рендеринг**: Використання DrawTexture() для спрайтів, DrawRectangle() для UI
- **Введення**: CheckCollisionPointRec() для кліків мишею, IsKeyPressed() для клавіатури
- **Звук**: LoadSound() та PlaySound() для звукових ефектів
- **Текстури**: LoadTexture() для спрайтів будівель та юнітів

## Моделі даних

### Характеристики фракцій

#### Рим
- **Стартові ресурси**: 200 їжі, 100 грошей
- **Особливість моралі**: 
  - Без їжі: втеча (combat_modifier = 0.0)
  - Без грошей: громадянський обов'язок (combat_modifier = 0.8)
- **Будівлі**: Преторій, Квесторій, Контуберній, Тенторіум

#### Карфаген  
- **Стартові ресурси**: 150 їжі, 200 грошей
- **Особливість моралі**:
  - Без грошей: повстання найманців (revolt = true, combat_modifier = 0.8)
  - Без їжі: голод (combat_modifier *= 0.8, стекується з повстанням)
- **Пасивний дохід**: +2 гроші/хвилину від Головного шатра
- **Будівлі**: Головне шатро, Шатер лівійців (3 рівні), Загін коней, Сховище їжі, Шатер купця

## Властивості коректності

*Властивість - це характеристика або поведінка, яка повинна залишатися істинною для всіх валідних виконань системи - по суті, формальне твердження про те, що система повинна робити. Властивості служать мостом між специфікаціями, зрозумілими людині, та гарантіями коректності, які можна перевірити машиною.*

### Властивість 1: Розміщення Обозу створює правильну будівлю
*Для будь-якого* Обозу та валідної позиції на карті, розміщення Обозу повинно створювати Преторій для Риму або Головне шатро для Карфагену відповідно до фракції
**Validates: Requirements 1.2**

### Властивість 2: Розкладання Обозу видаляє його та створює ресурси  
*Для будь-якого* Обозу, після розкладання юніт Обозу повинен зникнути, а створена будівля повинна мати початкові ресурси фракції
**Validates: Requirements 1.3**

### Властивість 3: Збір ресурсів рабами
*Для будь-якого* раба з завданням збору ресурсів, швидкість збору повинна дорівнювати базовій швидкості завдання помноженій на 0.7 (ефективність 70%)
**Validates: Requirements 2.3, 2.4, 2.5, 2.6**

### Властивість 4: Обмін на ринку
*Для будь-якої* операції обміну на ринку, 2 одиниці їжі повинні обмінюватися рівно на 1 одиницю грошей
**Validates: Requirements 2.7**

### Властивість 5: Характеристики будівель при створенні
*Для будь-якої* новоствореної будівлі, її характеристики (вартість, час будівництва, місткість) повинні відповідати специфікації типу будівлі
**Validates: Requirements 3.2, 3.3, 3.4, 4.2, 4.3, 4.4, 4.5, 4.6, 4.7**

### Властивість 6: Характеристики юнітів при створенні  
*Для будь-якого* новоствореного юніта, його характеристики (HP, атака, захист, швидкість) повинні точно відповідати специфікації типу юніта
**Validates: Requirements 5.1, 5.2, 5.3, 5.4, 6.1, 6.2, 6.3, 6.4, 6.5**

### Властивість 7: Мораль Риму при нестачі їжі
*Для будь-якого* римського юніта, коли у фракції закінчується їжа, юніт повинен мати fleeing = true та combat_modifier = 0.0
**Validates: Requirements 7.1**

### Властивість 8: Стекування дебафів Карфагену
*Для будь-якого* карфагенського юніта, коли у фракції немає грошей і немає їжі, combat_modifier повинен дорівнювати 0.8 * 0.8 = 0.64
**Validates: Requirements 7.5**

### Властивість 9: Формула розрахунку урону
*Для будь-якої* атаки між юнітами, урон повинен розраховуватися як (атака_нападника - захист_захисника) * модифікатор_моралі_фракції_нападника
**Validates: Requirements 9.1**

### Властивість 10: Умови перемоги через знищення HQ
*Для будь-якої* гри, знищення ворожого Преторію або Головного шатра повинно призводити до перемоги гравця
**Validates: Requirements 10.1**

## Обробка помилок

### Стратегія обробки помилок
1. **Валідація введення**: Перевірка всіх дій гравця перед виконанням
2. **Graceful degradation**: Система продовжує працювати навіть при частковій втраті функціональності  
3. **Логування**: Запис критичних помилок для налагодження
4. **Відновлення стану**: Автоматичне відновлення після збоїв

### Типи помилок та їх обробка

#### Помилки розміщення
- **Неприйнятна територія**: Показати повідомлення та заборонити розміщення
- **Перекриття з існуючими об'єктами**: Підсвітити конфлікт та запропонувати альтернативи
- **Недостатньо ресурсів**: Показати вартість та поточні ресурси

#### Помилки бойової системи  
- **Недійсна ціль**: Ігнорувати команду атаки
- **Юніт поза дальністю**: Автоматично наблизитися до цілі
- **Мертвий юніт**: Видалити з активних юнітів та оновити UI

#### Помилки ресурсів
- **Переповнення сховища**: Зупинити збір ресурсів та показати повідомлення
- **Від'ємні ресурси**: Встановити мінімум 0 та перерахувати стан фракції

## Стратегія тестування

### Подвійний підхід до тестування
- **Юніт-тести**: Перевіряють конкретні приклади, граничні випадки та умови помилок
- **Property-based тести**: Перевіряють універсальні властивості на багатьох згенерованих входах
- Обидва типи тестів є взаємодоповнюючими та необхідними для повного покриття

### Конфігурація Property-Based Testing
- **Бібліотека**: Використання Catch2 з генераторами для C++
- **Мінімум ітерацій**: 100 ітерацій на кожен property-тест
- **Теги тестів**: Формат **Feature: 2d-strategy-game, Property {номер}: {текст властивості}**

### Баланс юніт-тестування
- Юніт-тести корисні для конкретних прикладів та граничних випадків
- Уникати надмірної кількості юніт-тестів - property-based тести покривають багато входів
- Юніт-тести повинні фокусуватися на:
  - Конкретних прикладах, що демонструють правильну поведінку
  - Точках інтеграції між компонентами  
  - Граничних випадках та умовах помилок
- Property-тести повинні фокусуватися на:
  - Універсальних властивостях для всіх входів
  - Всебічному покритті входів через рандомізацію

### Приклади тестів

#### Property-Based Test
```cpp
TEST_CASE("Feature: 2d-strategy-game, Property 3: Збір ресурсів рабами") {
    SECTION("Раби збирають ресурси з 70% ефективністю") {
        auto slave_task = GENERATE(HUNT, FISH, GATHER, FARM);
        auto base_rates = std::map<SlaveTask, int>{{HUNT, 3}, {FISH, 2}, {GATHER, 1}, {FARM, 4}};
        
        Slave slave;
        slave.task = slave_task;
        slave.efficiency = 0.7f;
        
        int collected = slave.collectResource();
        int expected = base_rates[slave_task] * 0.7f;
        
        REQUIRE(collected == expected);
    }
}
```

#### Unit Test  
```cpp
TEST_CASE("Початкові ресурси фракцій") {
    SECTION("Рим починає з 200 їжі та 100 грошей") {
        Game game;
        game.initializeFaction(ROME);
        
        REQUIRE(game.getFactionResources(ROME).food == 200);
        REQUIRE(game.getFactionResources(ROME).money == 100);
    }
}
```
