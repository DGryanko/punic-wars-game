# Дизайн виправлення позиціонування в ізометричній системі

## Огляд

Після завершення повної ізометричної адаптації виявлено три критичні проблеми з позиціонуванням об'єктів, які виникли через неповну адаптацію систем рендерингу та руху до нової ізометричної системи координат GridCoords. Цей документ описує технічний підхід до виправлення кожної проблеми з детальним аналізом першопричин та конкретними змінами в коді.

## Глосарій

- **Bug_Condition (C)**: Умова, що викликає баг - неправильне позиціонування об'єктів в ізометричній системі
- **Property (P)**: Бажана поведінка - точне вирівнювання об'єктів відносно ізометричної сітки
- **Preservation**: Існуюча функціональність, що має залишитися незмінною (рух юнітів, рендеринг тайлів, камера)
- **GridCoords**: Система координат сітки (row, col) для ізометричної мапи
- **ScreenCoords**: Екранні координати (x, y) в пікселях для рендерингу
- **texture_offset**: Зміщення текстури відносно базової позиції об'єкта
- **CoordinateConverter**: Клас для конвертації між GridCoords та ScreenCoords
- **PathfindingManager**: Система пошуку шляху з використанням NavigationGrid
- **NavigationGrid**: Сітка навігації для pathfinding (16x16 пікселів на клітинку)

## Деталі багів

### Проблема 1: Будівлі розташовуються повз тайлів

**Умова виникнення:**

```
FUNCTION isBugCondition_Buildings(building)
  INPUT: building of type Building
  OUTPUT: boolean
  
  RETURN building.texture_offset.x == -192 
         AND building.texture_offset.y == -112
         AND building.useDebugRendering == false
         AND visuallyMisalignedWithTiles(building)
END FUNCTION
```

**Приклади:**

- **Преторій (HQ_ROME)**: Текстура 384x224 пікселі, footprint 3x3 тайли. При texture_offset = {-192, -112} будівля візуально виходить за межі тайлів на ~50 пікселів зліва та ~30 пікселів зверху.
- **Контуберній (BARRACKS_ROME)**: Текстура 384x224 пікселі, footprint 2x2 тайли. Аналогічне зміщення призводить до виходу за межі мапи при розміщенні біля краю.
- **Квесторій (QUESTORIUM_ROME)**: Текстура 384x224 пікселі, footprint 2x2 тайли. Клік по будівлі не спрацьовує, оскільки getRect() використовує texture_offset для визначення області кліку.

### Проблема 2: Монети зміщені на панелі ресурсів

**Умова виникнення:**

```
FUNCTION isBugCondition_Coins(resourceDisplay)
  INPUT: resourceDisplay of type ResourceDisplay
  OUTPUT: boolean
  
  RETURN moneyTextPosition.x == panelX + 680
         AND foodTextPosition.x == panelX + 280
         AND abs(moneyTextPosition.x - foodTextPosition.x) == 400
         AND expectedOffset == 280  // Має бути як у їжі
END FUNCTION
```

**Приклади:**

- **Панель ресурсів**: При panelX = 500, їжа відображається на X = 780 (500 + 280), а монети на X = 1180 (500 + 680). Різниця 400 пікселів замість очікуваних 0 (обидва мають бути на однаковій відстані від своїх іконок).
- **Візуальний ефект**: Число монет зміщене на ~60 пікселів праворуч від іконки монети, тоді як число їжі правильно вирівняне з іконкою амфори.

### Проблема 3: Юніти ігнорують геометрію мапи

**Умова виникнення:**

```
FUNCTION isBugCondition_Pathfinding(unit, pathfindingManager)
  INPUT: unit of type Unit, pathfindingManager of type PathfindingManager
  OUTPUT: boolean
  
  RETURN unit.is_moving == true
         AND pathfindingManager.requestPath called with screen coordinates
         AND NOT pathfindingManager.requestPath called with grid coordinates
         AND unit.path shows straight lines instead of isometric grid alignment
END FUNCTION
```

**Приклади:**


- **Рух юніта до цілі**: Юніт на позиції (10, 10) отримує команду рухатися до (20, 20). Система викликає `pathfindingManager.requestPath(unitId, {screenX1, screenY1}, {screenX2, screenY2})`, де координати в пікселях. NavigationGrid конвертує їх у свою внутрішню сітку 16x16 пікселів, яка НЕ відповідає ізометричній сітці тайлів.
- **Debug лінії**: При увімкненому debug режимі шлях відображається як прямі лінії між waypoints, які не слідують ізометричній геометрії тайлів.
- **Застрягання юнітів**: Юніти часто застрягають на краях будівель, оскільки NavigationGrid використовує прямокутну сітку замість ізометричної.

## Очікувана поведінка

### Коректна поведінка для будівель

Коли будівля розміщується на тайлі, система ПОВИННА:
- Відображати будівлю точно на тайлі з правильним вирівнюванням відносно ізометричної сітки
- Центрувати текстуру будівлі відносно її footprint (займаних тайлів)
- Забезпечити, щоб область кліку (getRect()) точно відповідала візуальному розташуванню текстури
- Для footprint 3x3 тайли: центр текстури має бути на центральному тайлі (1, 1)
- Для footprint 2x2 тайли: центр текстури має бути між чотирма тайлами

### Коректна поведінка для монет

Коли відображається число монет на панелі ресурсів, система ПОВИННА:
- Вирівнювати текст на координаті X = panelX + 280 (аналогічно до їжі)
- Забезпечити однакову відстань між іконкою та числом для обох ресурсів
- Зберегти вертикальне вирівнювання (Y = 50)

### Коректна поведінка для pathfinding

Коли юніт отримує команду руху до grid координат, система ПОВИННА:
- Використовувати GridCoords для обчислення шляху
- Конвертувати GridCoords у координати NavigationGrid з урахуванням ізометричної геометрії
- Забезпечити, щоб шлях слідував ізометричній сітці тайлів
- Debug лінії повинні показувати відповідність між шляхом та тайлами

### Вимоги до збереження існуючої поведінки

**Незмінна поведінка:**
- Рух юнітів по прохідних тайлах має продовжувати працювати без колізій
- Будівлі мають продовжувати блокувати тайли для pathfinding
- Панель ресурсів має продовжувати відображати їжу коректно на позиції panelX + 280
- Юніти мають продовжувати коректно зупинятися при досягненні цільової позиції
- Камера (WASD, миша, зум) має продовжувати коректно працювати
- Тайли мапи мають продовжувати використовувати ізометричний рендеринг з правильним depth sorting

**Область впливу:**
Всі зміни, що НЕ стосуються виправлення трьох описаних проблем, мають залишитися незмінними. Це включає:
- Систему виробництва юнітів у будівлях
- Систему збору ресурсів рабами
- Бойову систему (атаки, HP, урон)
- AI поведінку ворожих юнітів
- Систему виділення юнітів та будівель

## Гіпотеза першопричин

### Проблема 1: Некоректний texture_offset для будівель

**Аналіз:**


У файлі `cpp/src/building.h` (рядки 90-150) всі будівлі мають однаковий `texture_offset = {-192, -112}`. Це значення було встановлене для центрування текстури 384x224 пікселі, але воно не враховує:

1. **Ізометричну геометрію**: В ізометричній проекції центр текстури має бути зміщений відносно базової точки (position) з урахуванням кута 30°.

2. **Різні footprints**: Будівлі з footprint 3x3 та 2x2 мають різні центри мас в ізометричній проекції, але використовують однаковий offset.

3. **Базова точка**: `getScreenPosition()` повертає екранні координати верхнього лівого кута footprint (position.row, position.col), але текстура має бути центрована відносно центру footprint.

**Математичний розрахунок правильного offset:**

Для ізометричної проекції з тайлом 64x32 пікселі:
- Footprint 3x3: центр на тайлі (1, 1) відносно position
- Зміщення центру в screen coords: `centerOffset = gridToScreen(1, 1) - gridToScreen(0, 0)`
- Для текстури 384x224: `texture_offset = centerOffset - {192, 112}` (половина розміру текстури)

Очікуваний результат: `texture_offset ≈ {-160, -96}` (замість {-192, -112})

**Підтвердження гіпотези:**
- Код у `building.h:90-150` встановлює однаковий offset для всіх будівель
- Метод `getRect()` (рядок 127) використовує `texture_offset` для визначення області кліку
- Метод `draw()` (рядок 177) використовує `texture_offset` для рендерингу спрайту

### Проблема 2: Помилка в координаті X для монет

**Аналіз:**

У файлі `cpp/src/resource_display.h` (рядок 36) координата X для монет встановлена як `panelX + 680`:

```cpp
DrawText(TextFormat("%d", availableMoney), (int)panelX + 680, 50, 28, {255, 255, 255, 255});
```

Тоді як для їжі (рядок 33) використовується `panelX + 280`:

```cpp
DrawText(TextFormat("%d", availableFood), (int)panelX + 280, 50, 28, {255, 255, 255, 255});
```

**Підтвердження гіпотези:**
- Різниця 400 пікселів (680 - 280 = 400) є явною помилкою
- Обидва ресурси мають однакову структуру: іконка + число
- Панель ресурсів симетрична, тому обидва числа мають бути на однаковій відстані від своїх іконок
- Правильне значення: `panelX + 280` (як у їжі)

### Проблема 3: Pathfinding використовує screen coords замість grid coords

**Аналіз:**

У файлі `cpp/src/main.cpp` (рядок 1537) виклик pathfinding використовує screen координати:

```cpp
Vector2 start = {(float)units[i].x, (float)units[i].y};
Vector2 goal = {(float)units[i].target_x, (float)units[i].target_y};
pathfindingManager.requestPath(i, start, goal, 1.0f);
```

Проблема в тому, що:

1. **NavigationGrid використовує прямокутну сітку**: У `pathfinding.h:44-50` метод `worldToGrid()` конвертує screen координати в прямокутну сітку 16x16 пікселів:
   ```cpp
   gridX = worldX / cellSize;  // cellSize = 16
   gridY = worldY / cellSize;
   ```

2. **Ізометрична геометрія ігнорується**: Ця конвертація НЕ враховує ізометричну проекцію, тому шлях обчислюється в прямокутній системі координат.

3. **Unit.moveTo() працює з GridCoords**: У `unit.h:217-224` метод `moveTo(GridCoords)` правильно використовує grid координати, але pathfinding викликається з screen coords.

**Підтвердження гіпотези:**
- `pathfinding.h:44-50`: `worldToGrid()` використовує просту ділення на cellSize
- `main.cpp:1537`: виклик з `units[i].x, units[i].y` (screen coords)
- `unit.h:217`: `moveTo(GridCoords)` - правильний API, але не використовується для pathfinding

## Властивості коректності

Property 1: Fault Condition - Будівлі точно вирівняні з тайлами

_For any_ будівлі, де texture_offset був некоректним ({-192, -112}), виправлена система SHALL відображати будівлю з правильним вирівнюванням відносно її footprint, забезпечуючи, що текстура центрована відносно займаних тайлів та область кліку точно відповідає візуальному розташуванню.

**Validates: Requirements 2.1, 2.2**

Property 2: Fault Condition - Монети правильно вирівняні

_For any_ відображення панелі ресурсів, виправлена система SHALL відображати число монет на координаті X = panelX + 280, забезпечуючи однакову відстань між іконкою та числом для обох ресурсів (їжі та монет).

**Validates: Requirements 2.3**

Property 3: Fault Condition - Pathfinding використовує grid координати

_For any_ команди руху юніта, виправлена система SHALL використовувати GridCoords для обчислення шляху, конвертуючи їх у координати NavigationGrid з урахуванням ізометричної геометрії, забезпечуючи, що шлях слідує ізометричній сітці тайлів.

**Validates: Requirements 2.4, 2.5**

Property 4: Preservation - Існуюча функціональність незмінна

_For any_ функціональності, що НЕ стосується трьох описаних проблем (рух юнітів, блокування тайлів будівлями, відображення їжі, зупинка юнітів, камера, рендеринг тайлів), виправлена система SHALL виробляти точно такий самий результат, як оригінальна система, зберігаючи всю існуючу поведінку.

**Validates: Requirements 3.1, 3.2, 3.3, 3.4, 3.5, 3.6**

## Стратегія тестування

### Підхід до валідації

Стратегія тестування слідує двофазному підходу: спочатку виявити контрприклади, що демонструють баг на невиправленому коді, потім перевірити, що виправлення працює коректно та зберігає існуючу поведінку.

### Дослідницька перевірка умов багів

**Мета**: Виявити контрприклади, що демонструють баги ПЕРЕД реалізацією виправлення. Підтвердити або спростувати аналіз першопричин. Якщо спростуємо, потрібно буде переглянути гіпотези.

**План тестування**: Написати тести, що симулюють розміщення будівель, відображення панелі ресурсів та рух юнітів. Запустити ці тести на НЕВИПРАВЛЕНОМУ коді для спостереження помилок та розуміння першопричин.

**Тестові випадки**:

1. **Тест розміщення Преторію**: Розмістити HQ_ROME на позиції (10, 10) та перевірити візуальне вирівнювання з тайлами (провалиться на невиправленому коді - будівля зміщена)

2. **Тест кліку по будівлі**: Клікнути на центр текстури будівлі та перевірити, що `isClicked()` повертає true (може провалитися на невиправленому коді - область кліку зміщена)

3. **Тест відображення монет**: Відобразити панель ресурсів та виміряти відстань між іконкою монети та числом (провалиться на невиправленому коді - зміщення 400 пікселів)

4. **Тест руху юніта по ізометричній сітці**: Відправити юніта з (5, 5) до (15, 15) та перевірити, що шлях слідує ізометричній геометрії (провалиться на невиправленому коді - прямі лінії)

**Очікувані контрприклади**:
- Будівлі візуально зміщені відносно тайлів на ~50 пікселів
- Клік по будівлі не спрацьовує або спрацьовує в неправильному місці
- Число монет зміщене на 400 пікселів праворуч
- Шлях юніта показує прямі лінії замість слідування ізометричній сітці
- Можливі причини: некоректний texture_offset, помилка в координаті X, використання screen coords замість grid coords

### Перевірка виправлення

**Мета**: Перевірити, що для всіх входів, де умова багу виконується, виправлена функція виробляє очікувану поведінку.

**Псевдокод:**
```
FOR ALL building WHERE isBugCondition_Buildings(building) DO
  result := building.draw_fixed()
  ASSERT visuallyAlignedWithTiles(result)
  ASSERT clickAreaMatchesTexture(result)
END FOR

FOR ALL resourceDisplay WHERE isBugCondition_Coins(resourceDisplay) DO
  result := resourceDisplay.draw_fixed()
  ASSERT moneyTextPosition.x == panelX + 280
END FOR

FOR ALL unit WHERE isBugCondition_Pathfinding(unit) DO
  result := pathfindingManager.requestPath_fixed(unit.gridPosition, unit.targetGridPosition)
  ASSERT pathFollowsIsometricGrid(result)
END FOR
```

### Перевірка збереження існуючої поведінки

**Мета**: Перевірити, що для всіх входів, де умова багу НЕ виконується, виправлена функція виробляє той самий результат, що й оригінальна функція.

**Псевдокод:**
```
FOR ALL input WHERE NOT (isBugCondition_Buildings OR isBugCondition_Coins OR isBugCondition_Pathfinding) DO
  ASSERT originalFunction(input) = fixedFunction(input)
END FOR
```

**Підхід до тестування**: Property-based testing рекомендується для перевірки збереження, оскільки:
- Автоматично генерує багато тестових випадків по всій області входів
- Виявляє крайні випадки, які ручні unit тести можуть пропустити
- Надає сильні гарантії, що поведінка незмінна для всіх не-багових входів

**План тестування**: Спостерігати поведінку на НЕВИПРАВЛЕНОМУ коді спочатку для не-багових входів, потім написати property-based тести, що захоплюють цю поведінку.

**Тестові випадки**:

1. **Збереження руху юнітів**: Перевірити, що юніти продовжують рухатися по прохідних тайлах без колізій після виправлення pathfinding

2. **Збереження блокування тайлів**: Перевірити, що будівлі продовжують блокувати тайли для pathfinding після зміни texture_offset

3. **Збереження відображення їжі**: Перевірити, що число їжі продовжує відображатися на panelX + 280 після виправлення монет

4. **Збереження зупинки юнітів**: Перевірити, що юніти продовжують коректно зупинятися при досягненні цільової позиції

5. **Збереження камери**: Перевірити, що WASD, миша та зум продовжують коректно працювати

6. **Збереження рендерингу тайлів**: Перевірити, що тайли продовжують використовувати ізометричний рендеринг з правильним depth sorting

- Генерувати випадкові позиції будівель та перевіряти візуальне вирівнювання з тайлами
- Генерувати випадкові значення ресурсів та перевіряти правильне відображення на панелі
- Генерувати випадкові команди руху юнітів та перевіряти, що шлях слідує ізометричній геометрії
- Тестувати збереження існуючої поведінки для багатьох сценаріїв

### Інтеграційні тести
