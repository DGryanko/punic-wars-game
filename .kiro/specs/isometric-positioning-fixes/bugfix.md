# Документ вимог для виправлення позиціонування

## Вступ

Після завершення повної ізометричної адаптації гри "Punic Wars: Castra" виявлено три критичні проблеми з позиціонуванням об'єктів:

1. Будівлі візуально розташовуються поза межами тайлів
2. Число монет на панелі ресурсів зміщене на ~60 пікселів праворуч
3. Юніти рухаються по прямих лініях замість слідування ізометричній геометрії мапи

Ці проблеми виникли через неповну адаптацію систем рендерингу та руху до нової ізометричної системи координат GridCoords.

## Аналіз бага

### Поточна поведінка (дефект)

1.1 КОЛИ будівля розміщується на тайлі ТОДІ система відображає будівлю зі зміщенням відносно тайлу, краї будівлі виходять за межі тайлів та межі мапи

1.2 КОЛИ будівля має footprint більше 1x1 тайлу ТОДІ візуальне розташування не відповідає займаним grid координатам

1.3 КОЛИ відображається число монет на панелі ресурсів ТОДІ текст зміщений на ~60 пікселів праворуч від очікуваної позиції (координата X = panelX + 680 замість panelX + 280)

1.4 КОЛИ юніт отримує команду руху до grid координат ТОДІ система pathfinding використовує screen координати замість grid координат, що призводить до руху по прямих лініях

1.5 КОЛИ юніт рухається по мапі ТОДІ шлях не враховує ізометричну геометрію, debug лінії показують невідповідність між шляхом та ізометричною сіткою

1.6 КОЛИ користувач клікає на будівлю ТОДІ система не реагує на клік, будівлі не стають інтерактивними

### Очікувана поведінка (коректна)

2.1 КОЛИ будівля розміщується на тайлі ТОДІ система ПОВИННА відображати будівлю точно на тайлі з правильним вирівнюванням відносно ізометричної сітки

2.2 КОЛИ будівля має footprint більше 1x1 тайлу ТОДІ система ПОВИННА відображати будівлю так, щоб вона займала точно вказану кількість тайлів без виходу за межі

2.3 КОЛИ відображається число монет на панелі ресурсів ТОДІ текст ПОВИНЕН бути вирівняний на координаті X = panelX + 280 (аналогічно до їжі)

2.4 КОЛИ юніт отримує команду руху до grid координат ТОДІ система pathfinding ПОВИННА використовувати grid координати для обчислення шляху з урахуванням ізометричної геометрії

2.5 КОЛИ юніт рухається по мапі ТОДІ шлях ПОВИНЕН слідувати ізометричній сітці, debug лінії повинні показувати відповідність між шляхом та тайлами

2.6 КОЛИ користувач клікає на будівлю ТОДІ система ПОВИННА реагувати на клік, відкривати панель будівлі з інформацією про HP та можливості найму юнітів

### Незмінна поведінка (запобігання регресії)

3.1 КОЛИ юніт рухається по прохідних тайлах ТОДІ система ПОВИННА ПРОДОВЖУВАТИ дозволяти рух без колізій

3.2 КОЛИ будівля блокує тайли ТОДІ система ПОВИННА ПРОДОВЖУВАТИ позначати ці тайли як непрохідні для pathfinding

3.3 КОЛИ відображається панель ресурсів ТОДІ число їжі ПОВИННО ПРОДОВЖУВАТИ відображатися коректно на позиції panelX + 280

3.4 КОЛИ юніт досягає цільової позиції ТОДІ система ПОВИННА ПРОДОВЖУВАТИ коректно зупиняти юніт

3.5 КОЛИ користувач використовує камеру (WASD, миша, зум) ТОДІ система ПОВИННА ПРОДОВЖУВАТИ коректно переміщувати та масштабувати вид

3.6 КОЛИ відображаються тайли мапи ТОДІ система ПОВИННА ПРОДОВЖУВАТИ використовувати ізометричний рендеринг з правильним depth sorting
