# Документ вимог: Генератор ізометричної тайлінгової мапи

## Вступ

Генератор ізометричної тайлінгової мапи - це система для процедурної генерації ігрових карт на основі ізометричних тайлів для гри "Пунічні війни: Табори". Система повинна створювати різноманітні, збалансовані карти з різними типами місцевості в ізометричній проекції, що підходять для стратегічного геймплею.

## Глосарій

- **IsometricTile (Ізометричний тайл)**: Базовий графічний елемент карти у формі ромба в ізометричній проекції
- **TileMap (Тайлова карта)**: Двовимірна сітка тайлів, що формує ігрову карту в ізометричній проекції
- **TerrainType (Тип місцевості)**: Категорія тайлу (трава, вода, пісок, дорога тощо)
- **Generator (Генератор)**: Система, що створює тайлову карту за заданими параметрами
- **Seed (Зерно)**: Числове значення для детермінованої генерації карти
- **Passability (Прохідність)**: Властивість тайлу, що визначає чи можуть юніти проходити через нього
- **ScreenCoordinates (Екранні координати)**: Координати (x, y) на екрані в пікселях
- **GridCoordinates (Координати сітки)**: Логічні координати (row, col) тайлу в двовимірному масиві
- **IsometricProjection (Ізометрична проекція)**: Метод відображення 3D об'єктів на 2D площині під кутом

## Вимоги

### Вимога 1: Базова структура ізометричної тайлової карти

**Історія користувача:** Як розробник, я хочу мати базову структуру для зберігання та управління ізометричною тайловою картою, щоб ефективно працювати з ігровим світом.

#### Критерії прийняття

1. КОЛИ створюється тайлова карта, ТО Система ПОВИННА ініціалізувати двовимірний масив заданого розміру (рядки x колонки)
2. КОЛИ запитується тайл за координатами сітки (row, col), ТО Система ПОВИННА повернути відповідний тип місцевості
3. КОЛИ координати виходять за межі карти, ТО Система ПОВИННА повернути помилку або значення за замовчуванням
4. КОЛИ встановлюється тайл за координатами сітки (row, col), ТО Система ПОВИННА оновити тип місцевості в цій позиції
5. КОЛИ запитується розмір карти, ТО Система ПОВИННА повернути ширину та висоту в тайлах
6. КОЛИ конвертуються екранні координати в координати сітки, ТО Система ПОВИННА використовувати ізометричну математику для правильного перетворення
7. КОЛИ конвертуються координати сітки в екранні координати, ТО Система ПОВИННА розраховувати позицію з урахуванням ізометричної проекції

### Вимога 2: Типи місцевості

**Історія користувача:** Як гравець, я хочу бачити різноманітні типи місцевості на карті, щоб мати тактичні можливості та візуальне різноманіття.

#### Критерії прийняття

1. КОЛИ генерується карта, ТО Система ПОВИННА підтримувати тип "трава" як базову прохідну місцевість
2. КОЛИ генерується карта, ТО Система ПОВИННА підтримувати тип "вода" як непрохідну місцевість
3. КОЛИ генерується карта, ТО Система ПОВИННА підтримувати тип "пісок" як прохідну місцевість
4. КОЛИ генерується карта, ТО Система ПОВИННА підтримувати тип "дорога" як прохідну місцевість зі збільшеною швидкістю
5. КОЛИ запитується прохідність тайлу, ТО Система ПОВИННА повернути булеве значення відповідно до типу місцевості
6. КОЛИ запитується модифікатор швидкості тайлу, ТО Система ПОВИННА повернути значення (1.0 для трави, 1.5 для дороги, 0.0 для води)

### Вимога 3: Процедурна генерація карти

**Історія користувача:** Як гравець, я хочу грати на різних картах кожного разу, щоб гра залишалася цікавою та непередбачуваною.

#### Критерії прийняття

1. КОЛИ викликається генератор з seed-значенням, ТО Система ПОВИННА створити детерміновану карту (однакове seed = однакова карта)
2. КОЛИ викликається генератор без seed-значення, ТО Система ПОВИННА створити випадкову карту з унікальним seed
3. КОЛИ генерується карта, ТО Система ПОВИННА заповнити всі тайли валідними типами місцевості
4. КОЛИ генерується карта, ТО Система ПОВИННА створити зв'язні області однакового типу місцевості (не поодинокі тайли)
5. КОЛИ генерується карта, ТО Система ПОВИННА забезпечити мінімум 60% прохідної місцевості для геймплею

### Вимога 4: Алгоритм шумової генерації

**Історія користувача:** Як розробник, я хочу використовувати шумову генерацію для створення природних ландшафтів, щоб карти виглядали реалістично.

#### Критерії прийняття

1. КОЛИ генерується карта, ТО Система ПОВИННА використовувати алгоритм Perlin Noise або Simplex Noise
2. КОЛИ обчислюється шумове значення для координат сітки (row, col), ТО Система ПОВИННА повернути значення в діапазоні [0.0, 1.0]
3. КОЛИ шумове значення менше 0.3, ТО Система ПОВИННА призначити тип "вода"
4. КОЛИ шумове значення від 0.3 до 0.5, ТО Система ПОВИННА призначити тип "пісок"
5. КОЛИ шумове значення від 0.5 до 0.9, ТО Система ПОВИННА призначити тип "трава"
6. КОЛИ шумове значення більше 0.9, ТО Система ПОВИННА призначити тип "дорога" (рідкісні тайли)

### Вимога 5: Розміщення стартових позицій

**Історія користувача:** Як гравець, я хочу мати справедливі стартові позиції для обох фракцій, щоб гра була збалансованою.

#### Критерії прийняття

1. КОЛИ генерується карта, ТО Система ПОВИННА визначити дві стартові позиції для Риму та Карфагену
2. КОЛИ визначаються стартові позиції, ТО відстань між ними ПОВИННА бути не менше 30% діагоналі карти
3. КОЛИ визначаються стартові позиції, ТО обидві позиції ПОВИННІ бути на прохідній місцевості
4. КОЛИ визначаються стартові позиції, ТО навколо кожної позиції ПОВИННА бути область 5x5 тайлів прохідної місцевості
5. КОЛИ визначаються стартові позиції, ТО Система ПОВИННА забезпечити рівний доступ до ресурсів (ліс, вода) для обох фракцій

### Вимога 6: Рендеринг ізометричної тайлової карти

**Історія користувача:** Як гравець, я хочу бачити візуалізовану карту з ізометричними текстурами, щоб орієнтуватися в ігровому світі.

#### Критерії прийняття

1. КОЛИ карта рендериться, ТО Система ПОВИННА відобразити кожен тайл в ізометричній проекції з правильним сортуванням (painter's algorithm)
2. КОЛИ карта рендериться, ТО Система ПОВИННА використовувати тайлсет з ізометричними тайлами та DrawTextureRec() для малювання
3. КОЛИ карта більша за екран, ТО Система ПОВИННА рендерити тільки видимі тайли для оптимізації (viewport culling)
4. КОЛИ гравець прокручує карту, ТО Система ПОВИННА оновити видимі тайли відповідно до позиції камери
5. КОЛИ тайл не має завантаженої текстури, ТО Система ПОВИННА відобразити кольоровий ромб відповідно до типу місцевості
6. КОЛИ завантажується тайлсет, ТО він ПОВИНЕН завантажуватися один раз на початку гри, а не кожен кадр
7. КОЛИ рендеряться тайли, ТО Система ПОВИННА малювати їх у правильному порядку: спочатку задні ряди, потім передні (від верху до низу, зліва направо)

### Вимога 7: Збереження та завантаження карти

**Історія користувача:** Як розробник, я хочу зберігати та завантажувати згенеровані карти, щоб використовувати їх у кампаніях та тестуванні.

#### Критерії прийняття

1. КОЛИ карта зберігається, ТО Система ПОВИННА серіалізувати дані карти у текстовий формат (JSON або CSV)
2. КОЛИ карта зберігається, ТО файл ПОВИНЕН містити розмір карти, seed-значення та масив типів місцевості
3. КОЛИ карта завантажується з файлу, ТО Система ПОВИННА десеріалізувати дані та відновити структуру карти
4. КОЛИ файл карти пошкоджений або відсутній, ТО Система ПОВИННА повернути помилку з описом проблеми
5. КОЛИ карта завантажується, ТО всі тайли ПОВИННІ мати валідні типи місцевості

### Вимога 8: Параметри генерації

**Історія користувача:** Як розробник, я хочу налаштовувати параметри генерації, щоб створювати карти різних типів для різних місій.

#### Критерії прийняття

1. КОЛИ задається параметр "water_level", ТО Система ПОВИННА регулювати кількість води на карті (0.0 = мало води, 1.0 = багато води)
2. КОЛИ задається параметр "forest_density", ТО Система ПОВИННА регулювати щільність лісів (0.0 = мало лісів, 1.0 = багато лісів)
3. КОЛИ задається параметр "mountain_coverage", ТО Система ПОВИННА регулювати кількість гір (0.0 = мало гір, 1.0 = багато гір)
4. КОЛИ задається параметр "noise_scale", ТО Система ПОВИННА регулювати масштаб шуму (менше = більші області, більше = дрібніші області)
5. КОЛИ параметри виходять за межі [0.0, 1.0], ТО Система ПОВИННА обмежити їх до валідного діапазону

### Вимога 9: Інтеграція з Raylib

**Історія користувача:** Як розробник, я хочу інтегрувати генератор ізометричної карти з Raylib, щоб використовувати його в існуючій грі.

#### Критерії прийняття

1. КОЛИ завантажується ізометричний тайлсет, ТО Система ПОВИННА використовувати LoadTexture() один раз на початку гри
2. КОЛИ рендериться ізометричний тайл, ТО Система ПОВИННА використовувати DrawTextureRec() для малювання частини тайлсету
3. КОЛИ звільняються ресурси, ТО Система ПОВИННА викликати UnloadTexture() для тайлсету при завершенні гри
4. КОЛИ перевіряється клік на тайл, ТО Система ПОВИННА використовувати GetMousePosition() та конвертувати екранні координати в координати сітки з урахуванням ізометричної проекції
5. КОЛИ карта рендериться, ТО Система ПОВИННА працювати з частотою 60 FPS для карт до 100x100 тайлів з viewport culling
6. КОЛИ використовується тайлсет, ТО всі ізометричні тайли ПОВИННІ бути розміщені в одному PNG файлі у вигляді сітки
7. КОЛИ конвертуються координати, ТО Система ПОВИННА використовувати формули ізометричної проекції: screenX = (col - row) * tileWidth/2, screenY = (col + row) * tileHeight/2

### Вимога 10: Оптимізація продуктивності

**Історія користувача:** Як гравець, я хочу плавну роботу гри навіть на великих картах, щоб не було затримок та лагів.

#### Критерії прийняття

1. КОЛИ генерується карта розміром 50x50 тайлів, ТО час генерації НЕ ПОВИНЕН перевищувати 100 мілісекунд
2. КОЛИ генерується карта розміром 100x100 тайлів, ТО час генерації НЕ ПОВИНЕН перевищувати 500 мілісекунд
3. КОЛИ рендериться карта, ТО Система ПОВИННА використовувати culling для відображення тільки видимих тайлів
4. КОЛИ карта завантажена в пам'ять, ТО розмір даних НЕ ПОВИНЕН перевищувати 1 МБ для карти 100x100 тайлів
5. КОЛИ запитується тайл за координатами, ТО час доступу ПОВИНЕН бути O(1) (константний час)
